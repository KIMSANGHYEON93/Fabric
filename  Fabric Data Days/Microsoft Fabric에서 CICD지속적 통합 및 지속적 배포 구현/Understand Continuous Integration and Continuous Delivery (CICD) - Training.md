## CI/CD(지속적 통합 및 지속적 배포) 이해

100 경험치

여러분과 팀원들이 각자 Fabric 환경의 각 부분을 개발하고 유지 관리하는 업무를 담당하는 경우, 개발 작업을 통합하고 특정 사전 프로덕션 환경에 변경 사항을 게시할 준비가 될 때까지 격리된 개발 환경에서 작업하는 것이 가장 좋습니다. 변경 사항을 게시할 준비가 되면 변경 사항이 기존 코드를 손상시키거나 다른 개발자의 변경 사항을 방해하지 않는지 확인해야 합니다. 또한 코드 변경 사항을 저장하고 문제가 발생할 경우 되돌릴 수 있도록 해야 합니다. Fabric에 내장된 지속적 통합(CI) 및 지속적 배포(CD) 기능이 이러한 작업을 용이하게 해줍니다.

지속적 통합(CI)과 지속적 배포(CD)는 여러 개발자의 코드 기여를 하나의 주요 코드베이스로 통합하는 프로세스입니다. 기여는 자주 커밋되며, 자동화된 프로세스를 통해 새 코드가 빌드되고 테스트됩니다. 코드는 지속적으로 운영 환경으로 이전되어 기능 개발 시간을 단축합니다.

### 지속적인 통합

개발자가 장기간 로컬 컴퓨터에서 별도의 코드 브랜치 작업을 하고 작업이 완료된 후에야 변경 사항을 기본 코드베이스에 병합하는 경우, 이후 개발 단계에서 발견될 수 있는 충돌과 버그가 발생할 가능성이 높아지고 사용자에게 기능을 제공하는 속도가 느려질 수 있습니다.

지속적 통합(CI)은 버그와 코드 오류를 방지하고 기능을 지속적으로 개발하고 출시할 수 있도록 도와줍니다. CI에서는 버전 관리 시스템의 공유 코드 브랜치 또는 트렁크에 코드를 자주 커밋하고, 병합된 후에는 빌드 프로세스와 자동화된 테스트를 통해 변경 사항을 검증합니다. 새 코드와 기존 코드 간의 충돌을 개발 프로세스 초기에 파악할 수 있어 수정이 더 쉽고 빠릅니다.

### 지속적인 배달

CI(Continuous Integration)가 완료되면 코드가 스테이징 환경에 배포되어 프로덕션 환경에 출시되기 전에 더욱 자동화된 테스트가 수행됩니다.

### 지속적인 배포

지속적인 배포는 자동화된 테스트를 통과한 후 구조화된 배포 단계를 거쳐 자동으로 업데이트를 프로덕션 환경에 릴리스하는 프로세스입니다.

## Fabric에서 CI/CD 사용

CI/CD를 사용하여 Fabric 항목의 수명 주기를 관리하는 것은 통합과 배포라는 두 부분으로 구성됩니다. 통합은 Git을 사용하여 구현됩니다. 배포는 Fabric 배포 파이프라인을 사용하여 구현됩니다. 배포 또는 통합 자동화는 Fabric REST API를 사용하여 구현됩니다.

- **Git:** 팀이 브랜치를 사용하여 협업할 수 있도록 지원하고 버전 관리 기능을 제공합니다. 증분 코드 변경 사항을 관리하고 코드 이력을 확인하는 데 도움이 됩니다.
- **배포 파이프라인:** 개발, 테스트, 프로덕션 등 다양한 환경에 코드 변경 사항을 적용할 수 있습니다.
- **Fabric REST API:** 자동화를 지원하고 CI/CD 프로세스를 프로그래밍 방식으로 관리할 수 있습니다.
