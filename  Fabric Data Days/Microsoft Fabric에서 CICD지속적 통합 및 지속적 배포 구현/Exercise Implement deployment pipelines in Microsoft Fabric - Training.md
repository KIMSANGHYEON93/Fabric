## 연습: Microsoft Fabric에서 배포 파이프라인 구현

완전한 100 경험치

- 20분

이제 Microsoft Fabric에서 CI/CD를 사용할 수 있는 기회입니다.

이 연습에서는 Fabric 배포 파이프라인을 사용하여 개발, 테스트, 프로덕션 등의 환경 간에 변경 사항을 복사하는 프로세스를 자동화하는 방법을 알아봅니다.

운동을 시작하고 지시를 따르세요.

---
1. Fabric의 CI/CD 프로세스에서 Git의 역할은 무엇입니까?

- Git은 다양한 환경에 코드 변경 사항을 배포하는 데 사용됩니다.
- Git은 코드 변경 사항을 자동으로 테스트하는 데 사용됩니다.
- **Git을 사용하면 팀이 브랜치를 사용하여 협업하고 버전 관리 기능을 활용할 수 있습니다. 증분 코드 변경 사항을 관리하고 코드 이력을 확인하는 데 도움이 됩니다. (Correct)**

2. Fabric 작업 공간을 Git 저장소에 연결하는 목적은 무엇입니까?

- 작업 공간에서 변경한 내용을 라이브 환경에 자동으로 배포합니다.
- **작업 공간과 Git 간의 콘텐츠를 동기화하여 동일한 콘텐츠를 보유하도록 합니다. (Correct)**
- 여러 개발자가 충돌 없이 동시에 동일한 코드에서 작업할 수 있도록 합니다.

3. Fabric에서 배포 파이프라인의 주요 기능은 무엇입니까?

- 배포 파이프라인은 Fabric에서 새로운 콘텐츠를 만드는 데 사용됩니다.
- 배포 파이프라인은 Fabric에서 오래된 콘텐츠를 삭제하는 데 사용됩니다.
- **배포 파이프라인은 개발, 테스트, 프로덕션 단계를 거쳐 콘텐츠가 이동하는 과정을 자동화합니다. (Correct)**

## 피드백

이번 모듈 평가는 **"CI/CD (지속적 통합 및 지속적 배포)"**와 **"ALM (애플리케이션 수명 주기 관리)"**에 대한 내용입니다.

과거 데이터 엔지니어링은 "파일 복사해서 운영 서버에 붙여넣기" 식의 주먹구구식 배포가 많았지만, Fabric은 개발자스러운 **Git 연동**과 **배포 파이프라인**을 통해 체계적인 버전 관리를 지원합니다. DP-700 시험에서 매우 중요한 "운영 프로세스" 영역입니다.

시니어의 관점에서 명확하게 해설해 드리겠습니다.

---

### 1. Fabric의 CI/CD 프로세스에서 Git의 역할은 무엇입니까?

## 1. 🎯 출제 의도 파악 (The Hook)

- **핵심 주제:** **VCS(버전 관리 시스템)**로서의 Git의 본질적인 역할을 묻습니다.
    
- **함정:** Git 자체가 배포(Deploy)나 테스트(Test)를 수행하는 '실행 도구'라고 착각하기 쉽습니다. Git은 '저장소'입니다.
    

## 2. ✅ 정답 및 논리적 추론 (The Logic)

- **정답: Git을 사용하면 팀이 브랜치를 사용하여 협업하고 버전 관리 기능을 활용할 수 있습니다. 증분 코드 변경 사항을 관리하고 코드 이력을 확인하는 데 도움이 됩니다.**
    
- **논리적 흐름:**
    
    1. **Git**은 소스 코드의 **변경 이력(History)**을 저장하고 추적하는 도구입니다.
        
    2. 여러 개발자가 **브랜치(Branch)**를 따서 각자 작업하고, 이를 병합(Merge)하는 과정을 통해 **협업**을 가능하게 합니다.
        
    3. CI/CD 프로세스에서 Git은 "진실의 원천(Source of Truth)" 역할을 하며, 실제 배포나 테스트는 Git의 변경 사항을 감지한 다른 도구(Pipeline 등)가 수행합니다.
        

## 3. ❌ 오답 분석 (The Distractors)

- **Git은 다양한 환경에 코드 변경 사항을 배포하는 데 사용됩니다:** 배포의 주체는 **Deployment Pipeline**이나 **Azure DevOps Pipelines**입니다. Git은 배포할 코드를 '제공'할 뿐입니다.
    
- **Git은 코드 변경 사항을 자동으로 테스트하는 데 사용됩니다:** 테스트는 별도의 테스트 프레임워크나 스크립트가 수행합니다.
    

## 4. 💡 시니어의 실무 한 마디 (Pro Tip)

- **필수 개념:** Fabric은 현재 **Azure DevOps** 및 **GitHub**와 연동됩니다. 현업에서는 "Main 브랜치에는 절대 직접 커밋하지 않는다(Pull Request 필수)"는 규칙을 반드시 세워야 사고를 막을 수 있습니다.
    

## 5. 📝 한 줄 요약 (Takeaway)

- **Git은 코드를 '배포'하는 트럭이 아니라, 코드를 안전하게 보관하고 관리하는 '금고'입니다.**
    

---

### 2. Fabric 작업 공간을 Git 저장소에 연결하는 목적은 무엇입니까?

## 1. 🎯 출제 의도 파악 (The Hook)

- **핵심 주제:** **Fabric Workspace**와 **Git Repo** 간의 **동기화(Sync)** 메커니즘을 묻습니다.
    
- 웹 UI(Fabric)에서 작업한 것과 코드 저장소(Git)의 상태를 어떻게 일치시킬 것인가에 대한 문제입니다.
    

## 2. ✅ 정답 및 논리적 추론 (The Logic)

- **정답: 작업 공간과 Git 간의 콘텐츠를 동기화하여 동일한 콘텐츠를 보유하도록 합니다.**
    
- **논리적 흐름:**
    
    1. Fabric 워크스페이스 설정에서 Git을 연결하면, 화면에 **"소스 제어(Source Control)"** 버튼이 생깁니다.
        
    2. 이 기능의 핵심 목적은 워크스페이스의 변경 사항을 Git에 커밋(Commit)하거나, 반대로 Git의 변경 사항을 워크스페이스로 업데이트(Update)하여 **양쪽의 상태를 동기화(Sync)**하는 것입니다.
        
    3. 이를 통해 웹에서 작업한 내용이 코드로 변환되어 저장소에 안전하게 백업됩니다.
        

## 3. ❌ 오답 분석 (The Distractors)

- **작업 공간에서 변경한 내용을 라이브 환경에 자동으로 배포합니다:** Git 연결만으로는 배포되지 않습니다. 배포는 **배포 파이프라인(Deployment Pipeline)**의 역할입니다.
    
- **여러 개발자가 충돌 없이...:** Git을 쓴다고 충돌(Conflict)이 마법처럼 사라지지 않습니다. 오히려 충돌을 감지하고 해결할 수 있게 해주는 것이죠. 연결의 주 목적은 '동기화'입니다.
    

## 4. 💡 시니어의 실무 한 마디 (Pro Tip)

- **운영 팁:** Git 연동을 하면 Fabric 아이템(Notebook, Report 등)이 **JSON/YAML** 형태의 텍스트 파일로 저장됩니다. 덕분에 실수로 삭제해도 Git에서 복구할 수 있는 강력한 **"실행 취소(Undo)"** 기능을 갖게 되는 셈입니다.
    

## 5. 📝 한 줄 요약 (Takeaway)

- **Git 연결은 '내 작업 공간'과 '코드 저장소'를 거울처럼 똑같이(Sync) 맞춰주는 기능입니다.**
    

---

### 3. Fabric에서 배포 파이프라인의 주요 기능은 무엇입니까?

## 1. 🎯 출제 의도 파악 (The Hook)

- **핵심 주제:** **릴리즈 관리(Release Management)**입니다.
    
- 개발(Dev) → 테스트(Test) → 운영(Prod)으로 이어지는 환경 간의 **이동(Movement)**을 담당하는 도구를 묻습니다.
    

## 2. ✅ 정답 및 논리적 추론 (The Logic)

- **정답: 배포 파이프라인은 개발, 테스트, 프로덕션 단계를 거쳐 콘텐츠가 이동하는 과정을 자동화합니다.**
    
- **논리적 흐름:**
    
    1. **배포 파이프라인(Deployment Pipelines)**은 Fabric 내에서 서로 다른 워크스페이스(예: Dev_WS, Test_WS, Prod_WS)를 단계(Stage)별로 연결합니다.
        
    2. 버튼 클릭 한 번으로 Dev에서 만든 리포트나 노트북을 Test 환경으로 **복제(Deploy)**하고, 두 환경 간의 차이점(Diff)을 시각적으로 비교해 줍니다.
        
    3. 이것이 바로 Fabric이 제공하는 CI/CD의 **CD(지속적 배포)** 영역입니다.
        

## 3. ❌ 오답 분석 (The Distractors)

- **새로운 콘텐츠를 만드는 데 사용됩니다:** 콘텐츠 생성은 각 아이템(리포트, 노트북 등)의 에디터에서 합니다. 파이프라인은 '배달'만 합니다.
    
- **오래된 콘텐츠를 삭제하는 데 사용됩니다:** 수명 주기 관리의 일부일 순 있지만, 삭제가 주요 기능은 아닙니다.
    

## 4. 💡 시니어의 실무 한 마디 (Pro Tip)

- **필수 기능:** 배포 파이프라인을 쓸 때 **"배포 규칙(Deployment Rules)"**을 꼭 설정해야 합니다. Dev 환경에서는 '개발용 DB'를 바라보고, Prod 환경으로 넘어가면 자동으로 '운영용 DB'를 바라보게 **매개변수를 갈아끼워주는 기능**입니다. 이거 안 하면 운영 서버에서 개발 데이터를 보는 대참사가 일어납니다.
    

## 5. 📝 한 줄 요약 (Takeaway)

- **배포 파이프라인은 코드를 개발계에서 운영계로 안전하게 실어 나르는 '이삿짐센터'입니다.**
