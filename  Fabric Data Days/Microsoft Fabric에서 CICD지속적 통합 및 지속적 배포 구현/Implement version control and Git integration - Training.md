## 버전 제어 및 Git 통합 구현

100 경험치

지속적인 통합을 지원하기 위해 코드 변경 사항을 공유 저장소에 병합하는 경우가 많습니다. 공유 저장소는 GitHub이나 Azure DevOps와 같은 버전 관리 시스템의 일부입니다. 버전 관리는 시간 경과에 따른 코드 변경 사항을 관리하는 방법입니다. 코드 수정 사항을 추적하고, 코드 개발에 협업적으로 기여하고, 필요한 경우 이전 버전의 코드로 되돌릴 수 있습니다.

GitHub와 Azure DevOps는 Fabric에서 지원되는 버전 제어 시스템입니다. 이러한 버전 제어 시스템을 사용하면 브랜치라고 하는 코드 저장소의 복사본을 만들 수 있습니다. 이 브랜치를 사용하여 팀 코드의 기본 버전과 별도로 자신의 코드를 작업할 수 있습니다. 제출할 변경 사항이 있으면 저장소에 **커밋 하고 변경 사항을 기본 코드 브랜치와 병합할 수 있습니다.**

Fabric에서는 버전 제어와의 통합이 작업 공간 수준에서 이루어집니다. 작업 공간 내에서 개발하는 항목의 버전을 관리할 수 있습니다.

## Git 저장소에 연결

Fabric 작업 공간은 라이브 항목에 액세스하는 공유 환경입니다. 작업 공간에서 직접 변경한 내용은 다른 모든 작업 공간 사용자에게 우선 적용되어 영향을 미칩니다. 공유된 라이브 작업 공간 외부의 격리된 작업 공간에서 개발하는 것이 가장 좋습니다. 보호된 작업 공간에서는 자신의 브랜치에 연결하여 라이브 작업 공간의 콘텐츠를 보호된 작업 공간으로 동기화한 다음, 변경 사항을 자신의 브랜치 또는 기본 브랜치에 다시 커밋할 수 있습니다.

1. **Git 저장소 설정:** Git 통합을 구현하는 첫 번째 단계는 GitHub 또는 Azure DevOps에 Git 저장소를 설정하는 것입니다. 저장소는 항목을 저장하고 관리하는 중앙 위치입니다.
2. **Fabric 작업 공간을 Git 저장소에 연결합니다.** 다음으로, 저장소에 연결하려는 작업 공간 내에서 작업 공간 설정의 **Git 통합** 옵션을 통해 저장소에 대한 연결을 설정합니다.
	![작업 공간과 Git 공급자 연결 인터페이스의 스크린샷입니다.](https://learn.microsoft.com/en-us/training/wwl-data-ai/implement-cicd-in-fabric/media/connect-git.png)
	작업 공간과 Git 공급자 연결 인터페이스의 스크린샷입니다.
	Git에 작업 공간을 연결하면 동기화할 **Git 저장소 브랜치를 생성하거나 기존 브랜치를 선택합니다. Fabric은 작업 공간과 Git 간의 콘텐츠를 동기화하여 동일한 콘텐츠를 유지합니다.**
	![Git 공급자 인터페이스에서 브랜치 선택의 스크린샷입니다.](https://learn.microsoft.com/en-us/training/wwl-data-ai/implement-cicd-in-fabric/media/branch-selection.png)
	Git 공급자 인터페이스에서 브랜치 선택의 스크린샷입니다.

## Fabric 작업 공간 및 Git 저장소를 커밋하고 업데이트합니다.

저장소에 연결하면 작업 공간에 Git 상태 열이 표시되어 원격 브랜치의 항목과 비교하여 작업 공간의 항목 동기화 상태를 나타냅니다.

소스 제어 아이콘은 작업 공간과 저장소 간에 차이가 있는 항목의 수를 보여줍니다.

작업 공간과 저장소를 동기화하려면:

- 작업 공간을 변경할 때 **소스 제어 창의** **변경 사항** 선택을 사용하여 Git 브랜치와 동기화합니다 .
- Git 브랜치에 새로운 커밋이 만들어지면 **소스 제어 창의** **업데이트** 선택을 사용하여 작업 공간과 동기화합니다 .
	![작업 공간에서 커밋되지 않은 변경 사항의 스크린샷입니다.](https://learn.microsoft.com/en-us/training/wwl-data-ai/implement-cicd-in-fabric/media/git-status.png)
	작업 공간에서 커밋되지 않은 변경 사항의 스크린샷입니다.

## 분기 시나리오

개발 작업 중 작업 공간을 변경하면 다른 모든 작업 공간 사용자에게 영향을 미치므로 공유 작업 공간 외부에서 격리된 상태로 작업하는 것이 가장 좋습니다. 개발 작업을 공유 작업 공간과 격리하려면 다음을 사용하여 개발할 수 있습니다.

- 분리된, 고립된 작업 공간
- 보고서 및 의미 모델을 위한 Power BI Desktop이나 Notebook을 위한 VS Code와 같은 클라이언트 도구.

두 시나리오 모두 기능 개발 작업은 메인 코드 브랜치 대신 전용 브랜치에서 진행해야 합니다. 이렇게 하면 여러 개발자가 메인 브랜치에 영향을 주지 않고 기능 작업을 쉽게 수행할 수 있습니다.

### 전용 브랜치를 생성하고 풀 리퀘스트를 발행하고 Git과 작업 공간을 동기화합니다.

다음 단계에 따라 전용 브랜치를 만들고 풀 리퀘스트를 발행하여 브랜치의 변경 사항을 메인 브랜치로 가져오세요.

별도의 격리된 작업 공간을 사용하여 개발하는 경우:

1. 이 페이지의 "Git 저장소에 연결" 섹션에 있는 지침에 따라 개발 작업 공간을 메인 브랜치에 연결합니다.
2. Fabric 웹 인터페이스에서 작업하는 개발자인 경우, **소스 제어** 와 **새 작업 공간으로 분기를** 선택하여 메인 브랜치에 연결된 개발 작업 공간에서 작업에 대한 격리된 브랜치를 만드세요. 브랜치의 이름을 지정하고 다른 작업 공간과 연결합니다. 새 작업 공간은 새로 만든 브랜치와 동기화되어 작업을 위한 격리된 작업 환경이 됩니다.
	![새로운 작업 공간과 지점을 만드는 스크린샷입니다.](https://learn.microsoft.com/en-us/training/wwl-data-ai/implement-cicd-in-fabric/media/branch-out-to-new-workspace.png)
	새로운 작업 공간과 지점을 만드는 스크린샷입니다.
3. 브랜치에서 변경 사항을 적용한 다음 Fabric의 **소스 제어 인터페이스를 통해 격리된 브랜치에 변경 사항을 커밋합니다.**
4. 그런 다음 *Git에서* **풀 리퀘스트(PR)** 를 만들어 격리된 브랜치의 변경 사항을 메인 브랜치로 가져옵니다.
5. *PR이 메인 브랜치에 병합되면 Git의 메인 브랜치가 업데이트됩니다. 공유* 개발 작업 공간을 열면 Git의 새 콘텐츠를 공유 개발 작업 공간과 동기화하라는 메시지가 표시됩니다.

클라이언트 도구를 사용하여 개발하는 경우, 프로세스는 Fabric 웹 인터페이스에서 개발하는 경우와 비슷합니다.

1. 이 페이지의 "Fabric 작업 공간을 Git 저장소에 연결" 섹션에 있는 지침에 따라 개발 작업 공간을 메인 브랜치에 연결합니다.
2. 로컬 머신에 저장소를 복제합니다.
3. Fabric에서 테스트할 준비가 되면 변경 사항을 원격 저장소에 푸시하세요. 격리된 브랜치를 별도의 작업 공간에 연결하여 변경 사항을 테스트하세요.
4. *Git에서* PR을 발행하여 변경 사항을 메인 브랜치에 병합합니다.
5. 메인 브랜치와 연결된 공유 작업 공간을 열면 저장소의 변경 사항을 작업 공간으로 동기화할지 묻는 메시지가 표시됩니다.
